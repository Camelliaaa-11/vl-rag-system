# 项目实践报告

## 一、项目实践内容

### 1. 项目背景与意义、需求分析与项目内容

本项目围绕“具备语音交互、记忆能力和视觉理解能力的机器人问答系统”展开。随着大语言模型和多模态模型的发展，机器人不再只是执行固定指令，而是可以通过自然语言与用户交流，并结合环境图像理解现实场景。本项目的目标是在现有机器人平台上搭建一套可运行的语音交互系统，使机器人能够听懂用户问题、调用大模型生成回答，并通过语音播报反馈给用户。

项目初期曾考虑展厅导览场景，但在实践过程中发现，展厅场景对展品知识库、讲解内容和固定环境依赖较强，不利于快速验证机器人通用交互能力。因此后期将测试重点调整为日常或实验室场景，使用杯子、鼠标、键盘、箱子、外套等日常物品，重点验证语音问答、图文问答、上下文连续性和纠错记忆等能力。

项目核心需求包括：语音输入转文本、文本或图文问题理解、大模型生成回复、TTS 语音合成、长期记忆存储与召回、前端调试展示，以及后续接入视觉指向识别能力。

### 2. 项目相关概念与技术

本项目涉及 ASR、LLM、VLM、TTS、RAG 和机器人系统集成等技术。ASR 用于将用户语音转为文本；LLM/VLM 用于生成自然语言回答，其中 VLM 能够同时处理图片和文本；TTS 用于将模型回复转为语音播放。项目中选择讯飞 API 作为 TTS 方案，主要考虑其声音效果较好、接口成熟、接入成本较低 [1]。大模型方面，机器人主链路逐步切换到 Qwen Omni，用于支持纯文本与图文交互 [2]。

记忆模块采用短期对话历史和长期记忆结合的方式。短期记忆用于保存当前会话上下文，长期记忆用于记录用户传授的新知识、纠错信息和视觉摘要。例如用户纠正“箱子里的不是背包，是外套”后，系统应将该事实写入记忆，并在后续问答中召回。视觉方面，系统预留了摄像头图片、目标检测和 VLM 图文问答接口，使机器人能够围绕用户指向的现实物品回答问题。

在调研过程中，也参考了 ROS-LLM、低延迟 Pepper 机器人多模态交互、OpenVLA、StreamVLN 等相关工作 [4][5][6][7]，以及 RobotecAI RAI、Home Assistant LLM Vision、Qwen2.5-VL、InternVL、YOLOv8、GroundingDINO 和 Grounded-Segment-Anything 等开源项目 [8][9][10][11][12][13][14]。其中，ROS-LLM 主要关注 ROS 环境下将自然语言任务转换为机器人行为，并结合任务反馈进行反思；Pepper 机器人相关工作强调低延迟多模态交互和语音链路优化；OpenVLA 面向视觉-语言-动作模型，更多用于机器人动作控制；StreamVLN 关注连续视觉流中的语言导航和低延迟上下文建模。开源项目部分，RAI 提供 ROS2 机器人智能体框架参考，Home Assistant LLM Vision 提供家庭视觉理解场景参考，Qwen2.5-VL 和 InternVL 提供开源图文模型参考，YOLOv8、GroundingDINO 和 Grounded-Segment-Anything 则分别对应目标检测、开放词汇检测和分割能力。结合本项目阶段目标，当前重点放在“语音 + 图片 + 问答 + 记忆”的轻量多模态交互，而不是直接进行机器人动作控制或视觉导航。

### 3. 项目系统与功能设计

系统整体采用模块化设计，主要包括语音识别模块、机器人主处理模块、模型服务模块、记忆模块、语音合成模块、视觉输入模块和前端调试模块。语音识别模块负责监听用户语音并发布文本；`local_model_processor.py` 是机器人主链路，负责接收文本、读取图片、组织上下文、调用模型、写入记忆并触发 TTS；`tts_service.py` 封装讯飞语音合成接口；`memory` 目录负责短期历史、长期事实、事件和视觉摘要的存储与检索。

系统设计中重点考虑了“有图”和“无图”两种情况。如果有最新图片且图片时间与用户输入时间接近，则将图片和用户问题一起传给 VLM；如果没有图片或图片过期，则进行纯文本问答。前端调试页面用于展示机器人对话文本、图片快照、VLM 输入图片、视觉摘要和运行日志，方便调试机器人链路。

### 4. 项目关键技术与实现

项目关键实现之一是语音交互链路的串联。用户说话后，ASR 将文本写入指定话题，机器人主程序接收文本后完成处理。为了提升语音体验，系统对模型回复进行句子切分，并将 TTS 合成改为单线程排队，避免多句音频并发播放导致重复、打断或抢播。

第二个关键实现是图文输入判断。系统从固定目录读取最新图片，并使用图片文件修改时间作为快照时间。如果图片早于用户输入超过 10 秒，则认为图片与本轮语音不匹配，不传入 VLM，从而避免旧图片误导回答。对于图文问答，系统还在 prompt 中加入红色方框提示，要求模型重点识别红框内物体。

第三个关键实现是记忆机制。系统不仅保存对话历史，还能抽取用户明确传授的事实和纠正信息，写入长期记忆库。视觉摘要也可以并行生成并存储，减少对主回复延迟的影响。为了优化速度，记忆召回中同一问题只进行一次 embedding 编码，再复用向量查询不同记忆库，降低重复计算开销。

第四个关键实现是主链路场景剥离。随着项目从展厅导览转向日常/实验室场景，系统逐步将展品 RAG 从机器人主链路中剥离，避免普通问答无关触发展品知识库检索。这样既能减少延迟，也能让系统更聚焦于当前实验室环境中的物体识别、指代理解和自然对话。

### 5. 项目部署与测试

项目部署在 Ubuntu 机器人板卡上，主要通过 `start_all.sh` 启动完整链路，包括音频驱动、ASR 服务、前端后端、视觉相关服务和机器人主处理器。部署过程中重点处理了虚拟环境路径迁移、Python 依赖缺失、Linux shell 脚本兼容性、模型目录和数据目录路径等问题。由于项目涉及 ROS2 话题通信和相机、语音等外部服务，部署和调试时也参考了 ROS2 的节点、话题和启动文件组织方式 [15]。

测试分为语音交互测试、TTS 播放测试、记忆测试、图文输入测试和前端展示测试。语音交互测试验证用户语音能否正常转文本并生成回复；TTS 测试关注音频是否完整、是否重复、是否抢播；记忆测试验证用户纠错是否能被长期保存；图文测试验证图片是否能正确传入 VLM；前端测试用于观察实时对话、图片快照和日志。当前系统已能完成基础语音问答和部分图文链路验证，但视觉模块和真实场景适配仍需继续优化。

在速度测试中，系统通过分段日志记录记忆写入、记忆召回、模型生成、切句、TTS 合成和音频播放等阶段。通过这种方式可以定位延迟来源，并判断是模型接口、embedding、TTS 合成还是播放链路导致等待时间过长。后续优化也将继续以日志数据为依据，而不是凭主观感觉调整。

## 二、总结与感悟

通过本项目实践，我对大模型机器人系统的完整开发流程有了更深入的理解。相比单独调用一个大模型接口，真实机器人系统需要同时处理语音、视觉、记忆、播放、部署和调试等多个环节。任何一个模块不稳定，都会影响最终体验。

实践中我体会最深的是工程取舍的重要性。例如 TTS 最终选择讯飞 API，并不是因为它最前沿，而是因为它更稳定、声音效果更好、接入更省时。场景上从展厅转向实验室日常物品，也是因为后者更适合当前阶段验证机器人通用交互能力。

同时，我也认识到日志和可观测性非常重要。通过分段记录记忆召回、模型生成、TTS 合成等耗时，才能准确判断瓶颈，而不是凭感觉优化。后续如果继续完善项目，我会重点提升视觉理解的稳定性、减少端到端延迟，并进一步完善长期记忆和纠错学习能力。
