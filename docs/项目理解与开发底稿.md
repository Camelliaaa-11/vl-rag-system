# VL-RAG-System 项目理解与开发底稿

## 1. 文档目的

这份文档不是架构愿景说明，也不是对已有设计稿的复述，而是基于当前仓库代码、现有文档和模块实现状态整理出的“项目实际理解”。

用途有两个：

- 帮助后续开发时快速建立对项目真实现状的共识
- 区分“已经实现的能力”和“仍停留在设计文档中的规划项”

如果后续代码发生较大调整，这份文档也应同步更新。

---

## 2. 项目一句话概述

VL-RAG-System 是一个面向展厅导览机器人的多模态问答系统。它试图把展品知识库检索、语言模型回答、语音识别、语音播报和 ROS 机器人节点串成一条交互链路。

就当前代码现状而言，这个项目已经实现了一个可运行的 v1 主链路：

- 文本问题或语音输入进入系统
- 问题先经过 RAG 检索展品知识
- 再由 Ollama 驱动的大模型生成回答
- 在 Web 模式下把文本返回前端
- 在 ROS 模式下把文本切句后转为语音并驱动机器人播报

文档中描述的五层架构、记忆系统、多 Agent 协调、内省机制、用户画像等内容，大部分仍然属于规划方向，尚未完整落地为当前主链路的一部分。

## 3. 仓库整体结构理解

当前仓库中，真正有业务价值的核心目录主要有：

- `services/`
- `rag/`
- `agents/`
- `prompts/`
- `frontend/`
- `docs/`
- 根目录下的 `main.py`、`local_model_processor.py`、`config.py`

可以用下面的方式理解它们：

### 3.1 `main.py`

这是 Web 后端入口，使用 FastAPI 暴露最基础的接口：

- `GET /`
- `GET /health`
- `GET /status`
- `POST /chat`

其中真正有业务意义的是 `/chat`，它接收：

- 可选图片 `image`
- 可选问题 `question`

随后直接调用 `LLMService.generate_response_sync()` 返回回答。

这说明 Web 模式下没有复杂编排层，主流程很直接。

### 3.2 `local_model_processor.py`

这是 ROS 机器人模式下的核心总控节点，可以视为“机器人模式入口”。

它做的事情是：

- 初始化 `LLMService`
- 初始化 `TTSService`
- 订阅 `/asr/user_input`
- 从磁盘读取最近一帧图片 `latest.jpg`
- 调用 `LLMService.generate_response_stream()` 流式生成回答
- 按句切分调用 TTS
- 把生成的音频路径包装成 JSON，发布到 `/xunfei/tts_play`

从这个文件可以看出，机器人模式下的真正总控逻辑并不复杂，本质上还是“文本输入 -> LLMService -> TTS 输出”。

### 3.3 `config.py`

这是全局配置中心，负责：

- 路径管理
- 日志初始化
- 模型路径
- 向量库路径
- 语音服务配置
- 目录创建

当前项目里很多模块都直接依赖它。它是一个非常核心的基础文件。

### 3.4 `services/`

这是项目最重要的业务实现层。当前真正支撑项目运行的能力基本都在这里。

### 3.5 `rag/`

这是知识库构建和检索子系统。目前完成度较高，是项目最稳定的能力之一。

### 3.6 `agents/`

这是规划中的 Agent 层。当前已经有基础文件，但还没有成为实际主链路的一部分。

### 3.7 `prompts/`

这是提示词层，承载了“技心”这个角色的人设约束和回答风格控制。

### 3.8 `frontend/`

这是一个纯静态前端页面，没有前端构建工程，直接通过浏览器与 FastAPI 后端交互。

### 3.9 `docs/`

这里既有当前仍有参考价值的设计文档，也有一些偏历史交接或局部测试性质的文档。阅读时必须区分“现状”和“规划”。

## 4. 当前系统真实主链路

当前系统真正稳定可依赖的主链路，不在 AgentManager，也不在记忆系统，而是：

`入口 -> LLMService -> MuseumRetriever -> Ollama -> 输出`

### 4.1 Web 模式链路

Web 模式链路如下：

1. 前端页面 `frontend/index.html` 发起 `POST /chat`
2. `main.py` 接收图片和问题
3. `main.py` 调用 `LLMService.generate_response_sync()`
4. `LLMService` 先做 RAG 检索
5. `LLMService` 再把检索内容和提示词拼成最终输入
6. `LLMService` 调用 Ollama 模型生成回答
7. `main.py` 把回答封装成 JSON 返回前端

### 4.2 ROS 模式链路

ROS 模式链路如下：

1. `ASRMonitor` 从讯飞 AIUI 消息中解析文本
2. 文本发布到 `/asr/user_input`
3. `StreamingPopProcessor` 订阅这个话题
4. 总控从磁盘读取 `latest.jpg`
5. 总控调用 `LLMService.generate_response_stream()`
6. 获得流式文本后按句切分
7. 每句交给 `TTSService.generate_speech()`
8. 生成音频后发布播放命令到 `/xunfei/tts_play`

### 4.3 一个关键判断

不管是 Web 模式还是 ROS 模式，真正的推理核心都集中在 `LLMService`，其他层更多是在做输入输出包装。

这意味着：

- 如果后续要改回答质量、上下文拼装、检索注入方式，优先改 `LLMService`
- 如果后续要增强知识检索质量，优先改 `rag/retriever.py`
- 如果后续要接多轮记忆、情感共鸣、Agent 路由，也应先考虑如何接入 `LLMService` 主链路，而不是单独做一个悬空模块

## 5. LLMService 是当前第一核心模块

文件位置：`services/llm_service.py`

这是目前项目最核心的服务。

### 5.1 它当前负责什么

`LLMService` 当前负责：

- 初始化 RAG 检索器 `MuseumRetriever`
- 加载 `prompts/` 目录下的模板
- 组装系统提示词和问题提示词
- 拼接少量历史对话
- 调用 Ollama 模型
- 提供同步和流式两种兼容接口

### 5.2 它的工作流程

`generate_stream()` 的逻辑可以概括为：

1. 把 `prompts` 列表拼成一个问题字符串
2. 用问题去做 RAG 检索
3. 取系统提示词 `system_prompt.md`
4. 取问题模板 `identify_prompt.md`
5. 把检索结果填入问题模板
6. 如果有历史消息，则附加最近 4 条历史对话
7. 组装为 Ollama 的 `messages`
8. 调用 `ollama.chat(..., stream=True)`
9. 如果失败，回退到 `ollama.generate(..., stream=False)`

### 5.3 当前的实际特点

- 它是“提示词拼接式”服务，而不是复杂推理编排器
- 它的上下文注入主要依赖 RAG 文本和 prompt 模板
- 图片参数虽然经过接口传入，但当前并没有真正作为多模态消息送给 Ollama
- 所谓“多模态”在当前代码里更多体现在接口形态，而不是完整实现

### 5.4 对后续开发的意义

如果后续要做这些事情，优先入口都是这里：

- 真正接入图像理解
- 增强多轮记忆
- 接入共鸣引擎
- 引入 Agent 选择逻辑
- 控制输出风格
- 提升引用知识的稳定性

## 6. RAG 子系统理解

当前项目中，RAG 是最完整的能力模块之一。

核心文件：

- `rag/ingest.py`
- `rag/retriever.py`

### 6.1 数据来源

知识库数据来自：

- `data/raw_docs/艺术与科技展览数据.xlsx`

这个 Excel 文件是展品知识的原始来源。

### 6.2 入库过程

`rag/ingest.py` 的主要职责是：

- 读取 Excel
- 遍历各个 sheet
- 提取每个展品的字段
- 把字段拼成一个长文本
- 用本地 BGE 模型生成向量
- 写入 ChromaDB

向量库位置：

- `data/chroma_db_local_model`

集合名：

- `museum_local`

### 6.3 检索器 `MuseumRetriever`

`MuseumRetriever` 是当前线上实际使用的检索器。

它在初始化时会：

- 检查模型路径是否存在
- 检查向量库是否存在
- 初始化 Chroma PersistentClient
- 使用本地 `bge-small-zh-v1.5` 作为嵌入函数
- 获取 `museum_local` collection

### 6.4 检索返回内容的特点

`retrieve(query, top_k)` 返回的不是结构化 JSON，也不是原始 chunk 列表，而是一段已经格式化好的展品档案文本。

里面包含的信息通常有：

- 作品名称
- 设计作者
- 指导老师
- 类别标签
- 呈现形式
- 作品描述
- 设计动机
- 灵感来源
- 技术特点
- 所属展区

这意味着它更适合直接喂给 LLM 做上下文注入，但不适合做精细程序逻辑判断。

### 6.5 RAG 当前的边界

当前 RAG 的能力边界很明确：

- 它是一个文本展品知识检索系统
- 它不是一个完整的多路召回框架
- 它没有实现 rerank
- 它没有实现用户记忆检索
- 它没有实现混合检索
- 它没有实现引用片段级证据控制

也就是说，设计文档里提到的多路检索、画像召回、见解召回，目前基本都还没有真正进入现有代码主流程。

## 7. 语音与视觉子系统理解

### 7.1 ASR

文件位置：`services/asr_service.py`

这个文件同时承担两个职责：

- 一个服务类 `ASRService`
- 一个 ROS 节点类 `ASRMonitor`

#### 7.1.1 `ASRService`

这个类从设计上想支持：

- 麦克风录音
- 云端语音识别
- 本地语音识别
- 模式切换

但当前真实实现情况是：

- 录音和 wav 保存逻辑存在
- 云端识别和本地识别基本还是占位逻辑
- `recognize_cloud()` 和 `recognize_local()` 目前返回的是模拟文本

#### 7.1.2 `ASRMonitor`

这部分比 `ASRService` 更接近实际可用状态。

它负责：

- 订阅 `/xunfei/aiui_msg`
- 解析讯飞 AIUI 协议中的文本
- 去重过滤
- 保存识别文本到本地
- 发布到 `/asr/user_input`

所以从系统集成角度看，当前真正有用的不是“录音识别”，而是“把已有的讯飞消息转换成统一文本输入”。

### 7.2 Vision

文件位置：`services/vision_service.py`

视觉模块当前职责很单纯：

- 订阅 `virtual_camera/image_raw`
- 将 ROS `Image` 消息转成 OpenCV 图像
- 保存到磁盘

它当前没有完成视觉语义理解，更多是一个图像采集节点。

这也是当前架构里一个很重要的事实：

- 视觉服务只负责采图
- 图像理解并没有形成独立服务闭环
- 机器人总控拿到图像的方式是“读磁盘里的 latest.jpg”

### 7.3 TTS

文件位置：`services/tts_service.py`

TTS 当前使用讯飞 WebSocket 接口生成 MP3。

它有三类能力：

- 生成语音文件
- 合成并本地播放
- 播放已有音频数据

机器人模式里主要使用的是：

- `generate_speech(text, output_path)`

它由总控决定输出路径，然后再通过 ROS 话题发布播放命令。

## 8. 提示词系统理解

核心文件：

- `prompts/system_prompt.md`
- `prompts/identify_prompt.md`

### 8.1 `system_prompt.md`

这是“技心”角色的核心人格设定。

它规定了很多风格约束，例如：

- 角色定位
- 节奏感
- 说话视角
- 不用教导式语气
- 带有技术美学和沉静感
- 有特定禁区和表达边界

这份 prompt 决定了项目回答不像普通问答机器人，而更像“带审美气质的展厅导览角色”。

### 8.2 `identify_prompt.md`

这是把 RAG 检索结果和用户问题拼接起来的桥梁模板。

它会把：

- `context`
- `question`

放到一个更适合角色化表达的文本框架里，要求回答自然融合，不要平铺直叙。

### 8.3 提示词层的意义

当前系统的“角色一致性”主要不是靠复杂 Agent 逻辑实现的，而是靠这两份 prompt 实现的。

换句话说：

- Prompt 层在现阶段承担了大量“人格建模”和“表达风格控制”的职责
- 如果后续模型回答风格不稳定，先检查 prompt，不要第一时间怀疑 Agent 层

## 9. Agent 层真实状态

核心文件：

- `agents/base_agent.py`
- `agents/intro_agent.py`
- `agents/chat_agent.py`
- `agents/smalltalk_agent.py`
- `services/agent_manager.py`

### 9.1 目前属于“架子已搭，功能未落地”

这些文件表达了一个很清晰的规划方向：

- IntroAgent 负责讲解
- ChatAgent 负责深度对话
- SmallTalkAgent 负责闲聊
- AgentManager 负责路由和调度

但从当前实现看，它们还没有成为项目的真实核心逻辑。

### 9.2 当前存在的问题

#### 问题一：默认注册依赖缺失

`AgentManager` 在注册默认 agent 时引用了这些模块：

- `agents.scene_analyzer_agent`
- `agents.dialogue_agent`
- `agents.action_agent`

但当前仓库中并没有这些文件。

这说明 `AgentManager` 依赖的是一个未完成的目标结构。

#### 问题二：已存在的 Agent 只是占位实现

`intro_agent.py`、`chat_agent.py`、`smalltalk_agent.py` 里的 `process()` 基本只返回模板文本，没有真正接入：

- RAG
- 记忆系统
- 共鸣引擎
- 视觉分析

#### 问题三：异步接口没有正确使用

`BaseAgent.process()` 是 `async def`，但 `AgentManager.dispatch_task()` 里直接同步调用了 `self.agents[agent_name].process(task_data)`。

即使后续把 agent 补齐，这里也还需要重构调用方式。

### 9.3 结论

Agent 层目前不能被视为可直接扩展的成熟主干，而应视为“下一阶段架构预留层”。

如果后续要做功能，除非任务就是专门完善 Agent 层，否则优先不要把新逻辑堆到这里。

## 10. 共鸣引擎与人格化模块理解

文件位置：`services/resonance_engine.py`

这个模块从设计意图上是想做：

- 情感分析
- 人设配置加载
- 回复风格调节
- 人格化滤镜

### 10.1 当前实现状态

当前已经实现了：

- 读取 persona 配置文件
- 如果配置缺失则加载默认人格
- 通过简单词表计算情感分值
- 对部分固定短语做替换
- 根据情绪强弱做轻量后处理

### 10.2 当前局限

它目前的问题不是“完全没有实现”，而是“没有真正接入主链路”。

也就是说：

- 模块本身能实例化
- 一些方法可以独立调用
- 但 `LLMService` 并没有正式调用它来修正最终回答

### 10.3 开发上的意义

如果后续要把“技心”的人设再做稳一点，这个模块是可以接着扩展的。但正确方向不是单独增强它，而是把它和 `LLMService` 的输出后处理真正接起来。

## 11. 前端和接口现状

### 11.1 前端形态

`frontend/index.html` 是一个纯静态 HTML 页面，没有前端工程化构建。

它支持：

- 输入文本
- 上传图片
- 发送聊天请求
- 状态检测
- 简单重试
- 新话题清空

### 11.2 前端交互方式

它通过 `fetch()` 调用后端：

- 根路径做在线检测
- `/status` 做状态补充检测
- `/chat` 发送消息

### 11.3 当前 API 的真实范围

当前代码中真正实现的 API 很少，主要就是：

- `/`
- `/health`
- `/status`
- `/chat`

而 `docs/api_design.md` 中描述的大量 `/api/*` 接口，目前并未在代码中实现。

因此开发时必须记住：

- 文档中的 API 设计不等于当前系统已有接口
- 当前项目不是一个完整分层 API 服务，而更像一个以 `/chat` 为中心的问答后端

## 13. 当前已经实现的能力

基于当前代码，可以认为已经实现或基本实现的能力有：

- Web 端上传问题并获得回答
- FastAPI 后端提供基础聊天接口
- 基于 ChromaDB 和本地 BGE 的展品知识检索
- 通过 Ollama 调用本地模型生成回答
- Prompt 驱动的人设化回答风格
- ROS 模式下从 ASR 文本话题触发回答生成

- 图像采集节点持续落盘最新画面

## 14. 当前尚未真正落地或未接入主链路的能力

这些能力在文档中出现较多，但在当前项目中要么未实现，要么尚未成为主流程的一部分：

## 15. 当前最关键的技术债和风险点

后续开发最容易踩坑的点如下。

### 15.1 `AgentManager` 依赖未完成模块

`services/agent_manager.py` 中引用了不存在的 agent 文件，说明这部分还不能直接作为稳定扩展基础。

### 15.2 Agent 接口异步调用方式不正确

`BaseAgent.process()` 是异步方法，但 `AgentManager` 中按同步方式调用，后续补功能时必须先修这里。

### 15.3 `TTSService` 中使用了不存在的配置项

`services/tts_service.py` 中多处使用 `Config.TTS_OUTPUT_DIR`，但 `config.py` 当前定义的是 `AUDIO_OUT_DIR`。

这意味着：

- `generate_speech()` 可用，因为路径由外部传入
- 但 `speak()`、`synthesize()`、`play_audio()` 这些方法如果真正运行，可能会报配置错误

### 15.4 `rag/retriever.py` 中兼容类存在坏代码

`Retriever` 类中的 `get_stats()` 调用了不存在的方法 `get_collection_statistics()`。

此外，它对 `search()` 返回值的理解也不正确。说明这个兼容类并没有真正维护好。

### 15.5 `MuseumRetriever` 工程化程度不高

当前初始化过程中仍然大量使用：

- `print`
- `sys.exit`

这对于被服务导入的模块来说并不理想。后续如果要增强可测试性和服务健壮性，这里需要改为显式异常和日志方式。

### 15.6 配置里有默认密钥

当前 `config.py` 中直接写入了讯飞配置默认值，这在工程安全上是风险点，后续应迁移到环境变量或 `.env` 文件。

### 15.7 图像传递仍依赖磁盘

机器人模式通过 `latest.jpg` 传递图像，会引入：

- 磁盘 I/O
- 状态竞争
- 数据滞后风险

这在未来做性能优化时是一个很明显的重构点。

## 16. 后续开发时的建议工作原则

### 16.1 先认清“现状核心”

当前核心不是 Agent，也不是记忆系统，而是：

- `LLMService`
- `MuseumRetriever`
- `main.py`
- `local_model_processor.py`

后续大多数功能改造都应优先围绕这几个点展开。

### 16.2 先补主链，再补理想架构

如果要增加能力，优先顺序建议是：

1. 先强化 `LLMService` 的上下文构造和输出质量
2. 再增强 `RAG` 检索质量
3. 再把 `resonance_engine` 真正接入主链路
4. 最后再考虑把 Agent 层做成真正可调度的体系

### 16.3 明确分清“机器人模式”和“Web 模式”

这两个模式共享核心模型服务，但运行环境不同：

- Web 模式更适合本地调试
- ROS 模式更依赖外部节点、音频设备和 Ubuntu 环境

写代码时不要假设两边依赖一致。

### 16.4 尽量避免继续堆积“只在文档里成立”的模块

后续如果新增模块，最好满足至少一个条件：

- 已经接入主链路
- 至少有明确调用方
- 至少有测试或启动入口

否则仓库会继续出现“设计很多、实际没用上”的问题。

## 17. 我对这个项目的总体判断

当前这个项目最准确的定位不是“完整实现了五层智能架构的多 Agent 系统”，而是：

一个已经跑通的、以 RAG + LLM 为中心的机器人导览系统原型，并且在仓库中预埋了面向下一阶段演进的 Agent、记忆、共鸣、认知闭环接口。

换句话说：

- 它已经有一个可以工作的产品骨架
- 它还没有完全长成文档里理想的目标架构
- 它最适合的开发方式不是大而全重构，而是在现有主链路上逐步实化那些已经规划过的能力

## 18. 给后续写代码时的最简结论

如果只记住几条，请记住下面这些：

- 当前最核心模块是 `services/llm_service.py`
- 当前最稳定的能力是 `rag/` 检索
- Web 模式和 ROS 模式共用同一个回答核心
- Agent 层目前还不是可靠主干
- 设计文档里很多内容仍属于目标态，不是现状
- 后续改功能时，优先在主链路上做增量增强，而不是先铺新架构
