# 实验室事实记忆落地方案6.23

## 目标

在现有架构上增加一层“实验室事实记忆”，让机器人能够记住用户明确教给它的稳定信息，并在后续对话中通过向量检索召回，放入 LLM 的上下文中使用。

典型目标包括：

- “我们的教室在 211”后，再问“我们的教室在哪”能回答 211。
- “我的杯子叫 bue”后，再问“bue 是什么”能知道它是杯子。
- “那个不是杯子，是手机”这类纠错信息可以作为事实记忆保存，供后续回答参考。

本轮不修改用户识别逻辑，不做 YOLO 识别纠错，只负责把用户传授的新知识存下来并召回。

## 现有架构接入点

当前 `local_model_processor.py` 的主流程是：

1. 接收 ASR 文本。
2. `MemoryHub.record_turn()` 写入短期对话。
3. `MemoryHub.recall()` 检索短期历史、见解、事件、用户群体。
4. 将 `recall.combined_context` 放入 system prompt。
5. DeepSeek 生成回复，再进入 TTS。

新功能只插入第 2 步和第 3 步之间：

1. 写入短期对话。
2. 从用户原话中提取显式事实。
3. 将事实写入独立 Chroma 数据库。
4. 再执行 recall，这样当前轮和后续轮都可以检索到事实。

## 数据设计

新增 `LabFactEntry`：

- `fact_id`：事实 ID，按事实内容生成稳定 hash，避免完全相同事实反复写入。
- `user_id`：当前保留字段，默认 `anonymous`。
- `session_id`：来源会话。
- `category`：事实类型，如 `location`、`alias`、`definition`、`correction`。
- `subject`：事实主体，如“我们的教室”“我的杯子”“bue”。
- `predicate`：关系，如“在”“叫”“是”“纠正为”。
- `object`：事实对象，如“211”“bue”“手机”。
- `content`：给 LLM 使用的自然语言事实。
- `source`：来源，默认 `user_taught`。
- `confidence`：默认 `1.0`。
- `timestamp`：写入时间。

数据库位置：`data/memory/lab_fact_db`。

Chroma collection：`lab_fact_memory`。

Embedding 复用 `memory.insight.default_insight_embedding_fn()`，优先使用本地 Qwen embedding；如果模型不可用，则降级为关键词检索。

## 提取策略

第一版采用规则提取，不调用额外 LLM，避免增加延迟和费用。

会提取的表达：

- 位置类：`我们的教室在211`、`实验室在三楼`。
- 命名类：`我的杯子叫bue`、`这个机器人叫波普`。
- 定义类：`bue是我的杯子`、`211是我们的教室`。
- 纠错类：`那个不是杯子是手机`、`不是杯子，是手机`。

会跳过的表达：

- 明显问题句，如“教室在哪里”“bue是什么”。
- 太短、太长、只有闲聊语气的句子。
- 没有稳定事实关系的普通对话。

## 召回策略

`MemoryHub.recall()` 新增 `lab_facts` 结果，并在 `combined_context` 中加入：

```text
[实验室事实记忆]
- (location) 用户告诉我：我们的教室在211。
- (alias) 用户告诉我：我的杯子叫bue。
```

这一段放在“过往见解”和“过往事件”之前，因为它比总结性见解更像硬事实。

## 修改文件

- 新增 `memory/lab_fact.py`：事实模型、规则提取器、Chroma 归档和检索。
- 修改 `memory/memory_hub.py`：初始化事实记忆、写入事实、召回事实、拼接上下文。
- 修改 `memory/__init__.py`：导出新增类。
- 修改 `config.py`：增加事实记忆数据库路径和 collection 名称，并确保目录创建。
- 修改 `local_model_processor.py`：在用户话语写入短期记忆后调用 `observe_user_fact()`。

## 验证方式

基础验证：

```bash
python3 -m py_compile memory/lab_fact.py memory/memory_hub.py memory/__init__.py config.py local_model_processor.py
```

运行验证：

1. 启动服务。
2. 输入“我们的教室在211”。
3. 再输入“我们的教室在哪”。
4. 观察回复是否使用 211。

如果本地 embedding 模型不可用，仍可以通过关键词 fallback 检索到包含相同主体或对象的事实。
